# 第一作者邮箱搜索重构设计（LLM 主战场版）

## 背景

当前项目中，通讯作者邮箱命中率高，主要受益于 Crossref、EuropePMC、论文页等结构化或半结构化来源；第一作者邮箱命中率明显偏低，因为第一作者联系方式通常没有集中收录源，最终大量请求会落到 `email_finder.py` 的 Layer 6（LLM 联网搜索）。

现状中的主要问题不是“是否应该使用 LLM 联网搜索”，而是：

1. 第一作者场景下，前置确定性阶段常常无法提供足够证据，Layer 6 成为事实上的主战场。
2. Layer 6 当前仍然过于依赖 LLM 直接给出邮箱或薄弱 URL，导致“搜不到”或“搜错”都很常见。
3. 本地 URL 抓取仍以轻量 HTML 抓取为主，对国外高校站点、慢站、动态渲染站、目录页的处理能力不足。
4. 项目硬约束已经明确：**宁可少报也不能报错**，没有网页明文证据就应该返回“未找到”。

因此，这次改造的目标不是弱化 LLM，而是把 LLM 从“最终拍板邮箱的人”改造成“高质量证据页搜索器”，同时增强本地页面抓取和最终验收。

---

## 目标

1. 保持 LLM 联网搜索为第一作者邮箱搜索的主战场。
2. 提升第一作者邮箱搜索精度，优先保证“不误报”。
3. 允许牺牲部分召回率，但必须显著降低错误邮箱输出。
4. 将单篇 DOI 的一作搜索总耗时控制在 **3 分 30 秒以内**。
5. 将最终输出建立在“网页明确证据”上，而不是域名、姓名规则或模型猜测。

---

## 非目标

1. 不追求第一作者邮箱全命中。
2. 不依赖邮箱拼写规则推断最终邮箱。
3. 不在本次改造中重构通讯作者搜索主逻辑。
4. 不把 ResearchGate、Google Scholar、普通论文页直接视为最终证据页。

---

## 硬约束

1. **宁可漏报，也不能报错。**
2. 只要页面中没有明确出现邮箱或明确的变形邮箱，就必须返回“未找到”。
3. 不允许根据 `firstname.lastname@domain`、同机构域名、通讯作者域名等规律直接推断结果。
4. 变形邮箱（如 `[at]`、`(at)`、`dot`）可以视为明确证据，但必须来自页面实际文本，并在本地清洗后再使用。
5. LLM 可以参与搜索与文本识别，但最终结果必须经过本地抓取与验证。

---

## 总体设计

新的第一作者搜索结构保持“先低成本确定性来源，再进入 LLM 主战场”的思路，但将 Layer 6 改造成：

`LLM 搜索候选证据页 -> 本地筛选候选页 -> 浏览器级抓取页面 -> 提取邮箱/变形邮箱 -> 严格验收 -> 输出或未找到`

核心原则：

1. **宽进严出**：前段允许 LLM 搜到多种可能相关的证据页；后段只接受明确证据。
2. **LLM 负责找证据，不负责拍板邮箱。**
3. **页面证据优先于模型描述。**
4. **官网人员页、目录页、成员页优先于学术资料页与普通论文页。**

---

## 阶段设计

### 阶段 1：快速确定性来源

保留以下阶段，但加上严格时间预算：

1. `Layer1 ORCID`
   - 仅在作者有 ORCID 时运行
   - 低成本、低命中、可保留
2. `Layer2 Crossref`
   - 必须保留
   - 这是最快的确定性来源
3. `Layer3 论文页面抓取`
   - 保留，但限制入口和超时
   - 主要用于获取作者补充页、出版商页面中的直接线索
4. `Layer4 homepage 抓取`
   - 仅在已有明确 homepage 时运行
   - 不再作为泛化搜索的替代物

这些阶段的目的从“尽量覆盖所有可能性”改为“以低成本拿明确证据”。如果没有证据，不在这些阶段重复消耗时间。

### 阶段 2：LLM 联网搜索候选证据页

这是第一作者场景下的新主战场。

LLM 不直接输出最终邮箱，而是返回结构化候选页列表。搜索重点是：

1. 大学、医院、研究所官网个人页
2. faculty / staff / directory / profile 页面
3. lab / team / members / people 页面
4. 作者信息页、论文作者补充信息页
5. ORCID / Google Scholar / ResearchGate（辅助）

低优先级页面：

1. 普通论文落地页
2. 机构首页
3. 新闻页
4. 搜索结果页
5. 无法确认是目标作者本人的聚合页

### 阶段 3：候选页本地筛选

LLM 返回的不是直接结果，而是待抓取的候选页集合。本地要先做一次筛选，再决定抓哪些页面。

优先保留：

1. `is_official_domain = true` 的页面
2. `page_type` 为 `official_profile`、`faculty_page`、`staff_directory`、`lab_member` 的页面
3. `matched_signals` 同时包含 `name` 与 `org`
4. 与 `paper_title` 或 `doi` 明显相关的页面
5. 与 `cached_domain` 一致或接近的官网域名页面

降级或丢弃：

1. 搜索引擎结果页
2. 普通论文页优先级高于官网页的结果
3. 域名明显不是官网、也不是高质量学术资料页的页面
4. 解释中无法体现“这页像是目标作者本人页面”的结果

### 阶段 4：浏览器级抓取

当前项目的 `requests + cloudscraper + BeautifulSoup` 轻量抓取不足以覆盖一作场景中的慢站、动态站和目录页。因此要新增浏览器级抓取能力，并把它作为候选页抓取的优先方式。

浏览器级抓取的作用：

1. 处理 JS 渲染页面
2. 处理慢响应高校/医院站点
3. 抓取渲染后的 DOM 文本
4. 提高人员目录页、team 页、faculty 页的可提取性

现有轻量抓取仍保留，但定位为：

1. 快速通道
2. 浏览器抓取失败时的回退
3. 简单静态页面的低成本抓取

### 阶段 5：页面文本抽取与最终验收

抓到页面文本后，再判断：

1. 这页是不是目标作者本人页或强相关成员页
2. 页面中是否出现标准邮箱
3. 页面中是否出现变形邮箱
4. 姓名与机构是否强相关

只有满足以下条件之一才允许输出：

1. 页面中明确出现标准邮箱，且目标作者匹配成立
2. 页面中明确出现变形邮箱，且本地清洗后可还原为标准邮箱，且目标作者匹配成立

以下情况一律返回“未找到”：

1. 只有模型说有邮箱，但本地页面抓不到
2. 只有域名、没有邮箱正文
3. 只有姓名规律、没有邮箱正文
4. 页面中无法高置信确认目标作者

## Prompt 设计

### Prompt 1：联网搜索候选证据页

```text
你是学术作者联系信息检索助手。你的任务是联网搜索“目标第一作者”最可能出现联系方式的页面，并返回结构化候选结果。

【目标作者信息】
- 姓名：{name}
- 机构：{org}
- 论文标题：{paper_title}
- DOI：{doi}
- 已知通讯作者邮箱域名：@{cached_domain}

【任务目标】
请搜索与目标作者 {name} 最相关、最可能出现联系方式的页面。
重点找“目标作者本人”的人员页、成员页、目录页、个人页。
不要把普通论文页、搜索结果页、新闻页、聚合页当成优先结果。

【优先页面类型】
优先级从高到低：
1. 大学 / 医院 / 研究所官网个人页
2. faculty / staff / directory / profile 页面
3. lab / team / members / people 页面
4. 作者信息页、论文作者补充信息页
5. ORCID / Google Scholar / ResearchGate 等学术资料页

【低优先级页面】
以下页面即使相关，也只能低优先级返回：
- 普通论文落地页
- 论文摘要页
- 新闻报道页
- 机构首页
- 搜索结果页
- 无法确认是目标作者本人的聚合页

【强约束】
1. 不要猜测或构造邮箱
2. 不要返回搜索引擎结果页 URL
3. 不要把普通论文页排在官网人员页前面
4. 如果页面不能较高把握地说明是目标作者本人，降低排序
5. 如果没有高相关候选页，返回空列表
6. 如果有官网人员页，就优先返回官网人员页，而不是 Scholar、ResearchGate 或普通论文页

【排序原则】
优先返回同时满足以下条件的页面：
- 页面与目标作者姓名强相关
- 页面与目标机构强相关
- 页面与论文标题或 DOI 强相关
- 页面类型像人员页、成员页、目录页、个人页
- 页面可能出现联系方式
- 域名像学校、医院、研究所官网
- 如果域名与已知通讯作者邮箱域名一致或接近，可提高排序

【输出要求】
只返回纯 JSON，不要解释，不要 markdown 代码块。

{
  "candidates": [
    {
      "rank": 1,
      "url": "https://...",
      "page_type": "official_profile|faculty_page|staff_directory|lab_member|author_info|orcid|scholar|researchgate|other",
      "is_official_domain": true,
      "matched_signals": ["name", "org", "paper_title", "doi", "corr_domain"],
      "email_visible": "yes|no|unknown",
      "why_relevant": "一句话说明为什么这页像是目标作者本人页面",
      "confidence": 0.0
    }
  ]
}

最多返回 5 个候选页。
如果没有高相关结果，返回：
{"candidates":[]}
```

### Prompt 2：页面文本抽取

```text
你是网页证据抽取助手。下面是某个网页抓取后的文本，请判断它是否真的属于目标作者，并提取页面中明确出现的邮箱或变形邮箱。

【目标作者】
- 姓名：{name}
- 机构：{org}

【要求】
1. 只根据下面提供的页面文本判断
2. 不要补充网页中没有的内容
3. 不要根据姓名和域名规则猜邮箱
4. 可以识别页面中明确出现的变形邮箱，例如：
   - abc [at] xyz.edu
   - abc(at)xyz.edu
   - abc at xyz dot edu
5. 只有在页面文本里真的出现邮箱或变形邮箱时，才返回邮箱
6. 如果页面不是目标作者本人页面，明确指出

【输出格式】
只返回纯 JSON：

{
  "is_target_person": true,
  "person_name_found": "页面中识别到的人名",
  "org_found": "页面中识别到的机构",
  "email_found": true,
  "email_raw": "页面中原样出现的邮箱或变形邮箱，没有则为空",
  "email_normalized": "清洗后的标准邮箱，没有则为空",
  "evidence_snippet": "支持判断的短文本片段",
  "confidence": 0.0
}
```

## 要删除或降级的旧逻辑

### 删除

1. Layer 6 中“LLM 直接返回最终邮箱”的兜底路径
2. 任何没有本地页面证据、只靠 LLM 回答文本就输出邮箱的逻辑

### 降级

1. 当前 `Layer5` 中只试 `https://www.{base_domain}` 的浅层机构站点抓取
2. 无明确页面证据时的多轮重复 LLM 询问
3. 无区分地抓取所有候选 URL

---

## 时间预算

单篇 DOI 的一作搜索总上限：**210 秒**

建议预算：

1. 快速确定性来源：20 到 35 秒
2. LLM 搜索候选页：30 到 50 秒
3. 浏览器级抓取与验证：80 到 110 秒
4. 轻量回退抓取或补充验证：20 到 30 秒

预算原则：

1. 不在低收益阶段反复消耗时间
2. 把时间优先给候选页质量和页面抓取质量
3. 抓不到证据要尽快失败，不做猜测兜底

---

## 输出规则

最终输出仅允许两种结果：

1. 找到邮箱
   - 有网页明文证据
   - 页面和目标作者强相关
   - 邮箱为标准邮箱或可清洗的变形邮箱
2. 未找到
   - 没有页面明文证据
   - 页面抓取失败
   - 候选页不能高置信确认是目标作者
   - 只有推断，没有证据

系统不再输出“看起来像正确但无法证明”的邮箱。

---

## 验证计划

### 自动验证

1. 用现有 `test_batch.py` 或批量 DOI 集合复测
2. 记录：
   - 一作命中率
   - 一作错误率
   - 单篇平均耗时
   - 候选页类型分布
3. 对比改造前后，重点看错误邮箱是否下降

### 手动验证

重点挑这些 case：

1. 之前 “搜不到” 的一作样本
2. 之前 “LLM 可能搜错” 的样本
3. 国外高校站、医院站、个人主页站
4. 有变形邮箱的页面

人工检查：

1. 返回的候选页是否比旧逻辑更像“目标作者本人页”
2. 最终命中是否有页面证据
3. 没有证据时是否能正确返回“未找到”

---

## 实施顺序

1. 改写 Layer 6 Prompt，返回结构化候选页
2. 改造候选页筛选逻辑
3. 接入浏览器级抓取
4. 增加页面文本抽取与变形邮箱识别
5. 删除旧的“LLM 直接报邮箱”兜底
6. 重新跑批量测试并调参

---

## 预期结果

这次改造后，第一作者搜索的行为将从：

`前面几层没找到 -> 让 LLM 直接报邮箱 -> 容易搜不到或搜错`

改成：

`前面几层没找到 -> LLM 找高相关证据页 -> 本地强抓取 -> 页面有证据才输出`

预期收益：

1. 错报率显著下降
2. 候选页质量明显提高
3. 对国外慢站和动态页的处理能力更强
4. 最终结果更符合“宁可漏报也不能报错”的产品目标
